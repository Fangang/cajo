Inside the cajo framework.

 Every remotely accessible object in the framework is known as an Item. An application can host one, or more, items within its virtual machine. Items can call other items (local or remote) and an item can also be called by other items (local or remote). A special type of item is known as a Proxy. Proxies are items sent from one virtual machine, to execute in the context of another. This allows an application to be originally designed with all of its items local to its virtual machine, then evolve until each item exists in a separate virtual, or even physical machine; with no changes to the source code.

In the standard RMI framework, each remote object must have a unique "stub" compiled for it. This can quickly become burdensome, especially if the interaction between objects becomes highly diverse; it is completely impossible when method signatures are unknown at compile time. To address these problems, the framework defines only one functional interface; every item inherits the Invoke interface, i.e.: 
   Object invoke(String method, Object args);
The interface defines a single method, in which a method name is provided, is given an argument, and returns a result. This signature operates slightly differently from local Java method invocation in two distinct ways. First the return type must be of type Object or a derived type. This means an invocation cannot return a primitive type, such as int or char. Second, the argument type must also be of type Object, meaning it too cannot be a primitive type. However, just as with local invocations, both the argument and the return, can be of type Object[] indicating more than one object is being transferred, they can even be null, indicating no object transfer.

Aside from these minor differences, object and item interaction behave identically. Typically, a small wrapper is placed around an item reference to give it the identical semantics, and type-safety, of a local method invocation; consider these examples:

   void baker() {
      someItem.invoke("baker", null);
      // no argument or return
   }

   int baker(int charlie) {
      return ((Integer)someItem.invoke("baker", new Integer(charlie))).intValue();
      // primitive argument and return
   }

   Able baker(Charlie c) {
      return (Able)someItem.invoke("baker", c);
      // object argument and return
   }

   Able[] baker(Charlie c, Delta d) {
      return (Able[])someItem.invoke("baker", new Object[] { c, d });
      // multiple object argument and return
   }
Now when one object is invoking a method directly, or through an Invoke interface, to an item (local or remote) it is completely abstracted, and ultimately irrelevant to the invoking object.

The invocation of a method on a remote item using the Invoke interface is made possible using the Remote class. It utilizes a Java feature known as reflection. The Remote class can take any object, and place a remotely invokable wrapper around it. When other objects execute its invoke method, the wrapper uses reflection to find the method on its internal object, matching both the name and the argument types. If it is found, it executes that method, and returns the result, if any. If no matching method can be found on the internal object, a NoSuchMethodException is thrown.

This technique makes all public methods of the internal object remotely callable!

All objects are local. Machine boundaries, whether virtual or physical, are only illusions.
