Inside the cajo framework.

 Every object in the cajo framework is simply called an item. An item can exist within a Virtual Machine, and call other items, either local or remote. An item can also be called by other items, either local, or in other virtual, or even physical machines. An item can also be a proxy, meaning an item sent from one machine (virtual or physical) to execute in the runtime of another. This means that an application can be originally designed with all of its items local; then evolve until each item is in a separate virtual or physical machine; with no changes to the source code.

In the standard RMI framework, each remote object must have a "stub" compiled for it. This can quickly become burdensome, if the interaction between objects becomes highly diverse, and becomes completely impossible when the method signatures vary at runtime. To address this problem, the cajo framework defines only one functional interface.

Every cajo item inherits the Invoke interface: 
   Object invoke(String method, Object args);
 The interface defines a single method call, where a method name is provided, is given an argument, and returns a result. This signature operates slightly differently from a local Java method invocation, in two distinct ways. First the return type must be of type Object, or a derived type. This means an invocation cannot return a primitive type, such as int or char. Second, the argument type must also be of type Object, meaning it too cannot be a primitive type. However, just as with local invocations, both the argument and the return can be of type Object[], indicating more than one object is being transferred, or they can also be null, indicating no object transfer.

Aside from these minor differences, object and item interaction can behave identically. Typically a small wrapper is placed around an item reference, to give it the identical semantics, and type-safety, of a local method invocation.

Consider these examples:

   void baker() { someItem.invoke("baker", null); }

   Able baker(Charlie c) {
      return (Able)someItem.invoke("baker", c);
   }

   Able[] baker(Charlie c, Delta d) {
      return (Able[])someItem.invoke("baker", new Object[] { c, d });
   }
Now when an object is invoking a method directly, or through the Invoke interface to a local or remote object, or even to a proxy item, it is completely abstracted, and ultimately irrelevant to the invoking object.

The invocation of a method on a remote item, or proxy, is made possible through the Remote class. It makes use of a Java feature known as reflection. The Remote class takes any Object, and places a remotely invokable wrapper around it, thereby making it an item. When objects execute the Invoke interface, it uses reflection to lookup the method on its internal object, matching both the name and the argument types. When it is found it executes that method, and returns the result. Naturally, if no matching method can be found on the internal object, a NoSuchMethodException is thrown.

All objects are local. Whether virtual, or physical; machine barriers are just an illusion.
