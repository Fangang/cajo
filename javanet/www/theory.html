Inside the cajo framework.

 Every inter-operating object in the cajo framework is called an item . An item can exist within a Virtual Machine, and interact with other local items. An item can be called by remote items. An item can be a proxy, meaning an item sent from one Virtual Machine to execute in the runtime of another. This means that an application can be originally designed with all its objects local, and evolve until each object is in a separate Virtual Machine, with no changes to the source code.

In RMI, each class called on a remote object must have a "stub" compiled for it. This becomes burdensome, if the interaction between objects become highly diverse, and impossible when the method signatures vary at runtime. To address this issue; the cajo framework defines only one functional interface. Every cajo item implements the Invoke interface: 
   Object invoke(String method, Object args);
 The interface defines a single method call, whereby a method name is given, provided an argument, and returns an object. This signature differs from regular a Java method invocation in two distinct ways. First the return type must be of type Object, or a derived type. This means that an invocation cannot return a primitive type, such as int or char. Second, the argument type must also be of type Object, meaning it too cannot be a primitive type. However, both the argument, and the return can be of type Object[], indicating more than one object is being transferred, as well as null, indicating no object transfer.

Aside from these minor differences, object interaction can behave identically. Often, a small wrapper is placed around an item reference; to give it the identical semantics, and type-safety, of a local method invocation. Consider these examples: 
   void baker() { someItem.invoke("baker", null); } 
   Able baker(Charlie c) { return (Able)someItem.invoke("baker", c); } 
   Able[] baker(Charlie c, Delta d) { return (Able[])someItem.invoke("baker", new Object[] { c, d }); }
Now, whether an object is invoking a method directly, through the invoke interface to a local object, a remote object, or proxy, is completely hidden, and ultimately irrelevant to the external software.

The invocation of a method on a remote item or proxy is accomplished through the Remote class. It makes use of a Java feature known as reflection. The Remote class takes any Object, and places a remotely invokable wrapper around it. When items execute the invoke interface on a Remote object, it uses reflection to lookup the method on its internal object, matching both the name, and the argument types. When it is found, it executes that method call, and returns the result. Naturally, if no matching method can be found on the internal object, a NoSuchMethodException is thrown.
