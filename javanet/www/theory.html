Inside the cajo framework.

 Every object in the cajo framework is known as an Item. An item exists within a Virtual Machine, and call other items (local or remote). An item can also be called by other items. (local or remote) An item can also be a Proxy, meaning an item sent from one machine (virtual or physical), to execute in the runtime of another. This means that an application can be originally designed with all of its items local to the virtual machine, then evolve until each item is in a separate virtual, or even physical, machine with no changes to the source code.

In the standard RMI framework, each remote object must have a unique "stub" compiled for it. This can quickly become burdensome, if the interaction between objects becomes highly diverse, and is completely impossible when the method signatures vary at runtime. To address this problem, the cajo framework defines only one functional interface.

Every item inherits the Invoke interface: 
   Object invoke(String method, Object args);
 The interface defines a single method, in which a method name is provided, is given an argument, and returns a result. This signature operates slightly differently from a local Java method invocation in two distinct ways. First the return type must be of type Object, or a derived type. This means an invocation cannot return a primitive type, such as int or char. Second, the argument type must also be of type Object, meaning it too cannot be a primitive type. However, just as with local invocations, both the argument and return can be of type Object[], indicating more than one object is being transferred, they can also be null indicating no object transfer.

Aside from these minor differences, object and item interaction behave identically. Typically a small wrapper is placed around an item reference, to give it the identical semantics, and type-safety, of a local method invocation.

Consider these examples:

   void baker() { someItem.invoke("baker", null); }

   Able baker(Charlie c) {
      return (Able)someItem.invoke("baker", c);
   }

   Able[] baker(Charlie c, Delta d) {
      return (Able[])someItem.invoke("baker", new Object[] { c, d });
   }
Now, when an object is invoking a method directly, or through the Invoke interface; to a local or remote object, or even to a proxy; it is completely abstracted, and ultimately irrelevant to the invoking object.

The invocation of a method on a remote item, or proxy, is made possible using the Remote class. It makes use of a Java feature known as reflection. The Remote class takes any Object, and places a remotely invokable wrapper around it, thereby making it an item. When other objects execute the Invoke interface, the wrapper uses reflection to lookup the method on its internal object, matching both the name and the argument types. When it is found it executes that method, and returns the result, if any. If no matching method can be found on the internal object, a NoSuchMethodException is thrown. This technique effectively makes all public methods of the internal object remotely callable.

All objects are local. Machine boundaries, whether virtual or physical, are only illusions.
