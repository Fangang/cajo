Server items are the core of the framework.

A server is a Java Virtual Machine that makes one or more of its objects remotely accessible. A remotely accessible object is one which derives from the class BaseItem, meaning it implements the Invoke interface. The invoke method is implemented in the base class, and is declared as final. This is done to emphasize the fact that an Item is essentially just an object, as will be explained below.

The final invoke method is its fundamental public method, which actually makes it remotely callable, as defined by the paradigm. Remote methods call the invoke method, passing it both a method name, and a collection of arguments, possibly even none at all. The invoke implementation of the BaseItem class uses a technique called Java Reflection, whereby it finds a public method of its subclass exactly matching the name provided in the remote call, which accepts arguments of exactly the type, and in the exact order, provided by the caller. If there is a match, it will invoke that method on the object, and return the method result, if any. This effectively makes all public methods of subclasses of BaseItem remotely callable. When no matching method can be found, either in method name, or argument type and order, the invocation will result in a NoSuchMethodException being thrown.

Subclasses are free to implement public methods of any type, the only requirement is that another item on a different (or local) machine (virtual or physical) must know its method signatures; no differently than for traditional objects, with the one exception; there is no longer compile-time type checking. Also, due to a restriction in the Java Reflection mechanism, primitive types such as int or char, for example, cannot be sent as arguments or returned, rather, they must be sent in their object representations, in this case; Integer or Character.

If a legacy object has a long class tree, and cannot extend BaseItem, all is not lost! A wrapper item can be made that simply passes the invocations to its internal legacy object, returning the result, if any.

All invocations should be made on an item reference through its Invoke interface. While this technique isn't technically necessary when the item is local, meaning in the same Virtual Machine; it allows the called object to later be moved to a remote machine (virtual or physical) with no software changes! A useful design pattern is to simply place a wrapper around the Invoke object, where its methods pass the invocation on to the internal reference, for examples: 
   String toString() {
      // no arguments, object return ...
      return (String)invokeRef.invoke("toString", null);
   }
   boolean equals(Object o) {
      // single argument, primitive return...
      return ((Boolean)invokeRef.invoke("equals", o)).booleanValue());
   }
   void someMetnod(String a, String b, String c) {
      // multiple arguments, no return...
      invokeRef.invoke("someMethod", new Object{} { a, b, c });
   }
That's all there is to an item! To make an item remotely accessible, it is passed to the static method bind method. A single virtual machine can remote as many items as it wants. An item obtains a reference to a remote item using the static method getItem. As a de facto convention; the primary item served by a VM is typically bound under the name "main". If an application handles its items using their Invoke interface, it need not know, or even care, if the items are local or remote. However, should it be important to know, either the remoteness or locality of an item; this is easily tested in the following way: 

   if (ref instanceof RemoteInvoke) {
      // the reference is to a remote item ...
   } else {
      // the reference is to a local item ...
   }
This is the purpose for the creation of the RemoteInvoke interface; testing, as it is otherwise a bodyless interface.

A BaseItem can also define a MainThread object, which will be launched automatically, on the item's being made remote. The item's application specific functionality would go here. There is also a public member named thread, on which it can be interrupted, and on which the item's processing thread should periodically monitor, to see if the thread isInterrupted, and to perform an orderly shutdown.

Items are just objects, which can be also invoked by other (remote) objects.
