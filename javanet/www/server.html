Supporting cajo proxy item servers.

The CodebaseServer could be considered one of the world's smallest application server. It is completely customized to support the cajo framework. The entire server consists of a single class, yet it serves (no pun intended) two major purposes.

Codebase Service


When a proxy item is sent to another VM, it arrives solely as a serialized collection of many cooperating objects, and no code (i.e. class files). The receiving VM will first check in its local codebase to find class definitions for these objects, if it does, it will use them. However, as is often the case with proxies, it will not have the application specific classes needed by the received proxy. When this happens, the RMI framework steps in; it will call back to the item server, and request the class definitions. The primary use of the CodebaseServer is to manage this transfer, and it does it automatically and transparently. The necessary classes are provided to the remote host, loaded into its runtime, and the proxy then operates locally in the context of that VM.

All proxy-specific class files, and other files needed by the proxy, such as images or localized string tables, are packaged into a single jar file. The next topic, example, will illustrate a very easy way to do this. The jar file will be sent, as requested by remote proxy hosts, when reconstructing the proxy object in their runtimes.

You will need a codebase server any time you are sending proxies to another remote machine. You will also need one if you are sending application specific classes as arguments to a remote machine, even if you are not sending proxy items per se.

Client Service


If the previous topic on client loading seemed almost magical, this is the function of an application server. The previous discussion on codebase service could be provided by any general purpose webserver. What makes an application server special is knowing what to do with all that 'extra-stuff' appended to the end of the URL.

As explained previously, the CodebaseServer first checks if simply a jar file is being requested, if so, it fulfills the request, very generic. However, when the request is not for a jar file; here is where things get interesting.

The CodebaseServer will parse the URL to extract additional meaning from the request. Based on its content, one of three things will happen. First, and trivially, it may not understand the request, in this case, the request is rejected. However, if the request is understood; now one of two things will happen.

If no exclamation point is terminates the request, the CodebaseServer will interpret this as a request for the client to be run as an applet, inside the requesting browser. If an exclamation point is appended, this will be interpreted as a request for the client to be run as an application, on the desktop of the requesting host. The provided arguments themselves are used to configure the client for operation within the requester's VM, as explained in the client topic. Note: the client is also contained in a separate jar file. The CodebaseServer will also handle its transfer automatically.

The client loading process is as follows:
  1. The host requests the client
  2. The CodebaseServer returns instructions on how to do this
  3. The host requests the client codebase
  4. The CodebaseServer provides this
  5. The host asks the server item for its proxy
  6. The server item provides this
  7. The host requests the codebase needed by the proxy
  8. The CodebaseServer provides this
An elaborate process, to be sure, but all handled transparently behind the scenes. The code is all there, for the curious, but truth be told, like the Java API source, you need not read it, in order to use it effectively. This was only a high-level overview, for your edification, the mind-numbing technical details were intentionally omitted. If this already has your head spinning, don't worry about it; especially don't worry about the next section, simply go on to the conclusion.

CodebaseServer Sharing


OK, for you hot-shots who want to squeeze out every last ounce of performance, you probably already thought of this, but here it is anyway. Because the CodebaseServer performs the functionality of a generic http server, if you do not require the client application service functionality, you can share another item's CodebaseServer. Simply place your proxy jar file in the working directory of another CodebaseServer, and use the fully-qualified constructor of the ItemServer, to tell clients where to find the file. Note: The CodebaseServer need not even be in the same physical box as the server item. Just ponder the implications of that :-)

Conclusion


The CodebaseServer handles the low-level protocol details to make the cajo framework run transparently. It allows a single client application to be able to display any graphical proxy. Typically it is used in its application server capacity, solely by a single VM, to support its shared items. Its purpose is to bring the power of proxy items, and graphical proxy viewing to everybody, not just the rocket-scientists.

A framework-specific application server, to make proxy service simple.
