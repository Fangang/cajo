Free, simple, powerful: Transparent Distributed Computing
Overview of the cajo SDK
The cajo SDK provides a small yet detailed example of two important concepts: One is to easily develop Rich Internet Applications; which can be launched as full-screen Applets, and via WebStart. Second is to provide a powerful approach to dynamic distributed computing. The SDK is intended as a quick, yet comprehensive introduction into the application of cajo.

The simple architecture of the SDK is the foundation; no detailed knowledge of the inner workings of the cajo library is necessary, as it makes no significant impact on the source code. Everything is plain-old Java, no special tools are needed, nor IDE plug-ins, nor other 3rd party libraries; not even a servlet container. The optional remote Graphical User Interfaces are just plain-old JComponents: your Swing learning investment is preserved. The cajo library takes care of presenting them in the browser, and as remote desktop applications.
Architectural Overview
The are three fundamental components:
  1. Services
    A Service is simply an ordinary Java object, all of whose public methods are callable, from other remote objects. No special annotations are needed, nor XML files, it is fully transparent. A service object furnishes interesting functionality that other objects can use. A service may even use other remote service objects, to fulfill its functionality. A service is used to distribute functional modules across multiple JVMs for improved performance, scalability, and reliability.

  2. Controllers
    A Controller is an object requested by a remote object, to operate within its JVM. The controller arrives with a remote reference to its sending service, and has no reference to its receiving object, as it is strictly a servant. A controller is used to minimise network network traffic, and improve performance; particularly when a lot of invocations are needed on the sending service, or a lot of processing unique to the receiver is necessary. A service can furnish as many controllers as it wishes, including none, but typically only one.

  3. Proxies
    A Proxy is an object sent to a remote object, to operate within its JVM. In essence it is the logical inverse of a controller. A proxy arrives with a remote reference to its sending service, and a local reference to its receiving service. The sending service, in turn, receives a remote reference to its proxy. A proxy is used to minimise network traffic, and improve performance; particularly when a lot of invocations are needed on the receiving object, or the data sets being manipulated at the receiver are large. A service can provide as many types of proxies as it wishes, including none.

    In both cases, the receiving object views a controller or proxy as a local "intelligent stand-in" (aka agent) for the sending service. A service can, if necessary, interact with a remote service directly; using neither proxies nor controllers. 
    Sending Service <---------- Controller <--- Receiving Object (requested by)
Sending Service <--------------> Proxy ---> Receiving Object (sent to)
Ancillary components:
  • Server A Java object furnishing one or more services.
  • View A JComponent optionally furnished by a controller. It allows users to interact with a service, in a controlled manner.
A cajo object is provided to all components, this allows them to look up services needed for assistance dynamically, by interface.

Source Code Examples (excerpted from the SDK)
Example Service: 
package service;

public class Service extends util.AbstractService implements IService {
   public Service(String name) throws Exception {
      super("controller.Controller", name); // service specific controller
      // use cajo member object to connect to other services as needed
   }
   public String foo(String bar) { // service public method
      if (bar == null) throw new NullPointerException("null arg");
      return "foo invocation completed for " + bar;
   }
}
Example Proxy: 
package proxy;

public class Proxy extends util.AbstractProxy implements IProxy {
   // This is the reference on which the proxy will communicate with
   // its sending service.
   private final HomeService homeProxy;
   // This is the reference on which the proxy will communicate with
   // its receiving service.
   private LocalService localProxy;
   // This is the proxy's interface to its sending service.
   private interface HomeService {
      Object foo(String bar); // call it synchronously (blocking)
   // Future<Object> foo(String bar); // or call it asynchronously
   }
   // This is the proxy's interface to its receiving service.
   private interface LocalService {
   // String baz(String foo);
      Future<String> baz(String bar); // call it asynchronously
   }
   public Proxy(Object homeService) {
      super(homeService);
      homeProxy = proxy(homeService, HomeService.class);
   }
   public void init(Object localService, gnu.cajo.Cajo cajo) {
      super.init(localService, cajo);
      localProxy = proxy(localService, LocalService.class);
      System.out.println("Proxy arrived at service!");
      // use the cajo object to connect to other services as needed
   }
   // This method is invoked by the remote service
   public String bar(String baz) { return localProxy.baz(baz); }
}
Example Controller: 
package controller;

public class Controller extends util.AbstractController implements IController {
   // This is the reference on which the controller will communicate with
   // its sending service.
   private final Service serviceProxy;
   // This is the controller's interface to its sending service.
   private interface Service {
      String foo(String baz); // this method is just an example
   // Future<String> foo(String baz); // to call it asynchronously
   }
   public Controller(Object service) {
      super(service);
      serviceProxy = proxy(Service.class);
   }
   public void init(gnu.cajo.Cajo cajo) {
      super.init(cajo);
      System.out.println("Controller arrived at service!");
      // use the cajo object to connect to other services as needed
   }
   public util.AbstractView getView() {
      view = new view.View(); // controller specific view
      String result = baz("hello"); // for illustration
      ((View)view).center.display(result); // show in view
      // attach controller specific view action listeners here...
      return view;
   }
   public String baz(String bar) { // call remote service function
      return serviceProxy.foo(bar).toString();
   }
}
Design Advice
Construction:
Begin by subclassing AbstractService into MockService(s), providing simple/trivial implementations of the methods. This will provide a fully demonstrable system, in literally minutes. Next, subclass the MockService(s) into ActualService(s) with an eye toward always being able to demonstrate.

Testing:
For a distributed system, better than using unit tests; develop scripts of functional test scenarios.

Advanced:
Your applications will become increasingly flexible and powerful the more you use Interfaces, rather than Objects, for non-primitive argument and return types.
Conclusion
Security:
Accepting mobile code for proxies and controllers can be risky. A rogue object could hobble or even crash the server JVM simply by allocating lots of memory for instance. For this reason acceptance of mobile proxies and controllers is disabled by default. It should go without saying that an internet facing server should not accept mobile code. However inside the firewall, other servers can enable it by calling a single line of code. (An example of this is included in the SDK)

The cajo framework is just as intuitive and easy to use to as it looks. All it takes to get started is to download the very small SDK (120kB). The example source is openly licensed under the Apache Licence 2.0, and compiles and runs right out of the box. The autogenerated javadoc provides extremely detailed comments. Try the SDK as a starting template; modify the example classes as you wish, and you can have your own distributed system up and running Quickly.
