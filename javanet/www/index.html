Welcome to the cajo distributed object framework.

 The cajo project is a small, free library, enabling powerful, dynamic multi-machine interaction both within, and between, Java applications. It provides an easy-to-use, yet completely understandable framework, to simplify the use of RMI while at the same time harnessing its full potential. It has been successfully tested an deployed on J2ME, J2SE, and J2EE platforms.

Goals of the project:
  1. Establish a standard distributed paradigm, which becomes increasingly valuable with greater use.

  2. Encourage increased distributed inter-operation between applications and components.

  3. Foster development of scale-free component networks, to create flexible, resilient, and adaptive applications.
1. This framework allows an unlimited number of Java Virtual Machines to work together as one. The application code makes absolutely no distinction between local and remote invocations. Applications can evolve from all objects being on a single Virtual Machine, to all objects being on separate Virtual Machines, to all objects being on separate Physical Machines. The scalability comes with the framework, requiring no changes to the application's source code.

2. Unsigned applets, indistinguishable from applications in terms of interactivity, can be easily created. Transfer an application's entire user interface to any Java capable browser. Clearly separate the computational requirements of the model, view, and controller portions of an application, between different Virtual Machines.

3. The framework allows developers and applications to collaborate dynamically; linking their objects at will. This architecture can fulfill the real promise of Java: to make the network into one seamless computer. Soon applications can be built using optimized components of other distributed applications.

Getting started:

The framework consists of only 2 small packages; gnu.cajo.invoke (5 classes) defining the architectural foundation, and gnu.cajo.utils (8 classes) extending and applying the foundation to create the framework.

1. For those who want to understand the architecture of the framework first, review the invoke package, it's surprisingly small. Next move on to the utils package. This will make the provided example both readily understandable, and completely extensible.

2. For those who want to jump right in and learn by doing, there is an example package, consisting of just four small classes, completely exercising the framework. The project's root directory has batch scripts illustrating three fundamentally different ways to build the application.

Either way, after downloading all the source code, the next best step would be to check out the tutorial, then review the component articles on the right bar.
