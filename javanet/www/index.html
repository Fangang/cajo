Welcome to the cajo project:    IANA port 1198

The cajo project is a small, free library; enabling powerful dynamic multi-machine cooperation; both within, and between, Java applications. It provides a surprisingly easy to use, yet completely understandable framework, to simplify the use of RMI, while at the same time harnessing its full potential. It is being successfully deployed in Java EE, SE, and ME environments, all over the world. It is particularly easy to add to existing applications, as it requires no architectural redesign.

Project goals

1. Establish a free, simple, distributed paradigm; which becomes increasingly valuable with greater adoption.

2. Provide a clean, modular, and completely testable approach; to distributed application development.

3. Encourage increased distributed inter-operation; between developers, applications, and component libraries.

Major features

1. The framework allows any number of Java Virtual Machines to work together, literally as one. Application code makes no syntactic distinction, between local and remote objects; they can even be interchanged, at runtime. Applications can evolve from all objects being on a single Virtual Machine, to all objects being on separate Virtual Machines, to all objects being on separate Physical Machines. The scalability comes via the framework; it requires no changes to the application's code, or its structure.

2. Unsigned Applets and WebStart applications, indistinguishable from installed applications, can be created simply. Easily transfer an application's entire user interface, to any Java equipped client. Cleanly separate the components of the model, view, and controller portions of an application between multiple machines. Send both data and code, with complete transparency, and security. Again, all without source impact.

3. This framework allows both applications, and developers, to collaborate dynamically, linking their objects at will. This architecture can fulfill the true promise of Java: Turn the network into one seamless, evolving computer; link everything, from mainframes, to mobile phones. It can allow applications to be easily built using optimised components from other distributed applications. Remoting and linking of objects can be done with as little as one additional line of code.

Getting started

The framework consists of just two very small packages:
  • gnu.cajo.invoke (6 classes) defines the architectural core for clients
  • gnu.cajo.utils (8 classes) applies this core for the creation of servers
The project has no dependencies on any other frameworks, and works with all Java Runtime Environments, 1.2 and higher. This is critically important, to include JME / embedded devices. Co-operating Virtual Machines can be behind NAT, firewalls, even HTTP proxies. Best of all, the framework uses: no Bloody XMLTM nor Silly AnnotationsTM!

There are two general approaches to getting started:

1. For those who want to fully understand the implementation of this framework, first review the invoke package, it's surprisingly small. Next move on to the utils package. This will make the provided example code both immediately understandable, and quickly extensible. It's also an excellent way to learn about RMI, in detail. A theory page has been provided, especially for this group.

2. For those who want to jump right in, and learn by doing, there is the example. It consists of just four small classes, yet it completely exercises the entire framework, and demonstrates its capability. The project's root directory has batch files, and an ant script, illustrating three fundamentally different ways to build the example. The tutorial page, should be of particular interest to this group.

Either way, the best first step would be to read the short overview page. Also, many find trying the getting started, proxy, and scripting experiments quickly enlightening. JEE Developers: once you have familiarised yourself with the project, come see the migration framework, created just for you!

I hope you find it surprisingly easy to come up to speed. I also hope you find the new possibilities very exciting. The framework is clear and focused, and this project is not intent on senselessly making it larger. Rather, the primary goal is on promoting and assisting its adoption, into new and existing applications. Of course improvements and fixes are always welcome, from everyone! However, the interesting challenge is; all changes must be both backward and forward compatible; in order to maintain interoperability with currently running applications. E.g. consider the public telephone network: A 1920's-era rotary phone can call the newest satellite phone on the other side of the planet, and vice versa.
