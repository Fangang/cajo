Connecting to items independently of their IP address.

The Multicast class performs two complimentary functions. First, it allows an item to "broadcast" a remote reference to itself over the network. Second, it allows items to listen for; and capture, these reference broadcasts. Once captured, an item can contact the broadcasting item, if it wishes. These functions operate entirely independently of each other; meaning an item can broadcast without listening; and listen without broadcasting; it can even do both. If it is listening while it broadcasts, it will even capture its own reference.

The use of multicasting is application specific; it can be used to join a group of inter-operating items, or it can be used to announce a particular event. Essentially, it gives items an opportunity to connect to each other spontaneously, without having to know the IP address' of the listening items beforehand.

Note: for long-range multicasting to work, the network routers must be configured to pass UDP multicast packets. If they are not, the multicasting will work only within the subnet of origination. There are two parameters on which the items must agree in order to send and receive references between each other. An IP address and port number define a specific "channel". Changing either of these, switches to an entirely different channel. The UDP/IP address is different from a traditional TCP/IP address in that it doesn't identify a machine (and socket) rather it signals all machines that are a member of a "group".

An item can listen and broadcast on as many channels as it wants. Applications could range from performing all multicasts on a single channel, to using separate channels for each specific meaning. The framework defines a default channel, which can be a useful method for spontaneous inter-networking. To use a Star Trek metaphor; it could be likened to the 'hailing frequency' for contacting new and unknown items. An item broadcasts by passing a remoted reference to itself, or some other item, to the multicast's announce method.

An item can register itself with a multicast object to listen for these reference broadcasts. It does this by invoking multicast's listen method, passing in a reference to itself. When the multicast object receives a broadcast of a remote item reference, it will invoke the listener with a string of "multicast", and a reference to itself, as an argument. Logically, a listening object would either implement the following function: 
   public Object multicast(Multicast m);
 if it derived from BaseItem, otherwise it would test for the callback in its Invoke method implementation.

A multicast object has four public members:
  • iaddr The IP address of the broadcasting item.
  • item The remote reference to the broadcasting item.
  • address The address on which this reference was received.
  • port The port on which this reference was received.
The reason for calling the listener with a reference to the multicast object is to allow the possibility for a single listener to listen on multiple multicast objects. If the listener is listening to only one object, there is no chance of its multicast method being invoked reentrantly. However, if a listener is listening on more than one, the possibility exists for reentrant invocations.

A listening multicast object checks the return value of its listener's invocation. If it returns a null, it will continue listening; if the method returns a non-null value, it will stop listening. To start the object listening again, its listen method must be called, either with the same, or a different listener. The multicast's listen method can only monitor using one listening object at a time. This means that if the multicast object is actively listening, and its listen method is invoked again, it will throw an IllegalArgumentException.
